## 2. ⚠️ ข้อสังเกตสำคัญ — สเปคนี้ยัง "ไม่ครบ" สำหรับเว็บสมัครงานจริง

สเปคของลูกค้าเป็นแค่ **เว็บ "ประกาศ/ค้นหางาน" + CMS** เท่านั้น
**ไม่มีขั้นตอน "สมัครงานจริง" (Apply)** ซึ่งเป็นหัวใจของเว็บ careers สมัยใหม่ ควรเสนอเพิ่ม:

- ปุ่ม/ฟอร์มสมัครงาน + อัปโหลดเรซูเม่ (PDF)
- โปรไฟล์ผู้สมัคร / บัญชีผู้ใช้ / ประวัติการสมัคร
- อีเมลแจ้งเตือน "ได้รับใบสมัครแล้ว"
- หลังบ้านดูรายชื่อผู้สมัครต่อ 1 ตำแหน่ง (mini-ATS)

## 3. ฟังก์ชันมาตรฐานที่เว็บสมัยใหม่ "ควรมี" (เพิ่มเติมจากสเปค)

### ประสบการณ์ผู้ใช้ / การค้นหา
- ค้นหาแบบ real-time + **pagination + sort** (ใหม่สุด/เกี่ยวข้องสุด)
- บันทึกงานที่สนใจ (Save/Bookmark) + **Job Alert** (สมัครรับอีเมลเมื่อมีงานใหม่ตรงเงื่อนไข)
- แชร์งานลงโซเชียล / คัดลอกลิงก์
- Responsive มือถือ 100% + รองรับ dark mode

### SEO & การค้นเจอ
- **Structured Data `JobPosting` (schema.org)** → ขึ้น Google for Jobs ฟรี (สำคัญมากสำหรับเว็บหางาน)
- Sitemap / robots.txt / meta OG / URL อ่านง่าย
- Server-side rendering เพื่อ SEO

### กฎหมาย & ความปลอดภัย (จำเป็นในไทย)
- **PDPA**: หน้ายินยอม (consent) เก็บข้อมูลส่วนบุคคล + นโยบายความเป็นส่วนตัว + Cookie consent
- reCAPTCHA / rate limit กันบอทสแปมใบสมัคร
- HTTPS, เข้ารหัสข้อมูลผู้สมัคร, กำหนดอายุการเก็บข้อมูล
- WCAG (Accessibility) ขั้นพื้นฐาน

### หลังบ้าน (Admin)
- **Role-based access** (Super admin / HR / Editor) + Audit log
- Dashboard สรุปตัวเลข (จำนวนงานเปิด, ผู้สมัคร, ยอดเข้าชม)
- Rich text editor สำหรับ JD, จัดการหมวด/แผนก/สถานที่แบบ master data

### วิศวกรรม / ปฏิบัติการ
- Google Analytics 4 / conversion tracking
- CDN + caching รูปภาพ, image optimization
- CI/CD, backup ฐานข้อมูลอัตโนมัติ, monitoring/uptime alert
- Environment แยก non-prod / prod (ตรงกับที่สเปคระบุ UAT/PVT/Go-live)

## 4. Tech Stack — **ยึดตามสเปคของ Central (ตัดสินใจแล้ว)**

**ตัดสินใจใช้ stack ตามที่ Central ระบุ: Vue.js + Laravel + Java Spring Boot + MySQL + Docker**

**เหตุผล (สำคัญ):** โมเดลการดูแลระบบคือ **ปีแรกเราทำ MA → ปีถัดไป Central รับไปดูแลเอง**
ดังนั้นระบบต้องอยู่บน stack มาตรฐานที่ทีม IT ในบ้านของ Central รับช่วงต่อได้ ถ้าใช้ stack อื่นทีมเขาจะดูแลต่อไม่ได้ → กลายเป็นจุดอ่อน ไม่ใช่จุดขาย

**ผลที่ตามมาต่อวิธีทำงาน (design for handover):**
- เขียนโค้ดให้อ่านง่าย มี test + CI/CD, ไม่ใช้ dependency แปลกที่ทีมเขาดูแลยาก
- ส่งมอบ **source code + เอกสารสถาปัตยกรรม + คู่มือ deploy + KT session** ให้ทีม Central ช่วงปลายปี MA → เขียนเป็น deliverable ในใบเสนอราคา (จุดขายหลัก)
- ค่า MA คิด **1 ปี** เท่านั้น (ไม่มีรายได้ต่อเนื่องปีถัดไป) → margin ค่าพัฒนาต้องเผื่อให้พอ

**ข้อ clarify กับลูกค้า (ไม่แย้งตรงๆ):** stack มี back-end 2 ชั้น (Laravel + Spring Boot) — ขอความชัดเจนว่าต้องการแบ่งหน้าที่กันอย่างไร
เช่น Laravel = CMS/หน้าบ้าน, Spring Boot = core API / Store-Job (2.10) เผื่อ Central มีเหตุผลเชิงสถาปัตยกรรมองค์กรอยู่แล้ว

| ส่วนระบบ | เทคโนโลยี (ตามสเปค) | บทบาท (เสนอให้ชัด) |
|---|---|---|
| Front-end | Vue.js | หน้าบ้านผู้ใช้ (ค้นหา/ดูงาน) |
| CMS/หลังบ้าน | Laravel (PHP) | จัดการงาน/Banner/แบรนด์/กิจกรรม |
| Core API | Java Spring Boot | business logic + Store-Job API (2.10) + Spring Security |
| DB | MySQL | ฐานข้อมูลหลัก |
| Deploy | Docker (SSD 180GB/RAM 8/CPU 4) | non-prod + prod |

## 6. ราคา (เปรียบเทียบกับคู่แข่ง 120,000 บาท)

ใบเสนอราคาคู่แข่ง:
| รายการ | ราคา (บาท) |
|--------|-----------|
| ค่าพัฒนาระบบ (หน้าบ้าน + หลังบ้าน + API) | 90,000 |
| ค่าดูแลระบบ 1 ปี (MA) | 20,000 |
| Host/VM 1 ปี (SSD 180GB/RAM 8/CPU 4) | 10,000 |
| **รวม** | **120,000** |

> **ความเห็น:** 90,000 บาท สำหรับเว็บ careers 2 ภาษา + CMS เต็ม + API บน stack 3 เฟรมเวิร์ก (Vue+Laravel+Java) ถือว่า **ต่ำมาก** (≈ ราคาฟรีแลนซ์/ทีมเล็ก) แทบไม่รวม buffer สำหรับ Apply flow, SEO, PDPA และไม่มีต้นทุน handover/KT เราจึงตั้งราคาตามมูลค่างานจริง

> **หมายเหตุ:** ราคาด้านล่างคิดบน **stack ตามสเปคของ Central (Vue + Laravel + Spring Boot)** ตามที่ตัดสินใจแล้ว จึงรวมต้นทุนการดูแล 3 เฟรมเวิร์ก + งานส่งมอบให้ทีม Central ดูแลต่อ (source code + เอกสาร + KT) ไว้เรียบร้อย

### ทางเลือก A — ตรงสเปค (เท่าที่ลูกค้าขอ) เพื่อ "สู้ราคา"
| รายการ | ราคา (บาท) |
|--------|-----------|
| ค่าพัฒนา (3 ส่วนตามสเปค บน stack Central) | 140,000 – 180,000 |
| MA 1 ปี (พร้อม handover/KT ปลายปี) | 25,000 – 30,000 |
| Host/VM 1 ปี | 12,000 – 15,000 |
| **รวม** | **~180,000 – 225,000** |

### ทางเลือก B — สเปคครบเว็บสมัยใหม่ (รวม Apply/SEO/PDPA/Dashboard) *แนะนำ*
| รายการ | ราคา (บาท) |
|--------|-----------|
| ค่าพัฒนา (สเปค + ฟังก์ชันมาตรฐานข้อ 3 บน stack Central) | 230,000 – 300,000 |
| MA 1 ปี (พร้อม handover/KT ปลายปี) | 35,000 – 45,000 |
| Host/VM 1 ปี | 12,000 – 18,000 |
| **รวม** | **~280,000 – 360,000** |

> ทั้ง A และ B ใช้ stack ตามสเปคแล้ว จึงไม่มี "ทางเลือก stack เรา" อีกต่อไป — จุดแข่งขันย้ายไปที่ **คุณภาพ + security process + ความครบของฟังก์ชัน + handover ที่ทีม Central รับต่อได้จริง**

**ค่าใช้จ่ายที่ลูกค้ารับผิดชอบเอง (ไม่รวมในราคา):**
- Google Maps API (ตามการใช้จริง — Central Group เปิดบัญชีเอง ตามหมายเหตุ 5.1)
- Domain (careers.centralgroup.com — ลูกค้าต้อง unmap host เดิมก่อน)

**เงื่อนไขชำระเงินที่แนะนำ:** งวด 1 = 40–50% ตอนเริ่ม · งวด 2 = 40% ตอน UAT ผ่าน · งวด 3 = 10–20% ตอน Go-live (แทนที่จะ 50/50 แบบคู่แข่ง เพื่อลดความเสี่ยงกระแสเงินสด)

## 7. สิ่งที่ควรถามลูกค้าให้ชัดก่อนยืนราคา
1. **การแบ่งหน้าที่ Laravel vs Spring Boot** — อยากให้ back-end 2 ชั้นนี้ทำอะไรกันบ้าง (เผื่อมีมาตรฐานองค์กรอยู่แล้ว)
2. **ยืนยันโมเดล MA:** ปีแรกเราดูแล → ปีถัดไปทีม Central รับต่อเอง (มีผลต่อขอบเขต handover/KT)
3. ต้องมี **ขั้นตอนสมัครงานจริง (Apply + resume)** ไหม หรือแค่ประกาศงาน?
4. "ระบบสร้างงาน" ที่จะยิงเข้ามาทาง API (2.10) — สเปค/รูปแบบข้อมูลเป็นอย่างไร (REST/JSON? auth?)
5. จำนวนตำแหน่งงาน/ผู้เข้าชมโดยประมาณ (มีผลต่อ sizing ของ VM)
6. ต้องรองรับ PDPA / เก็บข้อมูลผู้สมัครตามข้อกำหนด compliance ระดับองค์กรไหม
7. ใช้ domain/subdomain เดิมหรือใหม่ และใครดูแล DNS
