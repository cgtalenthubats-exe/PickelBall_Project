# POS2U Integration — สถานะและแผน

> สถานะ: **รอ API doc จาก POS2U** — ยังไม่ลงมือออกแบบ schema/โค้ดจนกว่าจะได้ spec จริง เพื่อไม่ให้เสียงานถ้าสมมติฐานผิด

## การตัดสินใจที่ล็อกไว้แล้ว

**ระบบจะรับชำระเงินได้ 2 ทาง (ไม่ใช่แทนกัน):**

1. **ออนไลน์** — Stripe (PromptPay QR / บัตร) ตอนจองผ่านเว็บ/แอป → `confirmed` ทันที (ตามแผนเดิมใน `docs/brief-v2.md`)
2. **หน้าร้าน** — ผ่านเครื่อง **POS2U** ตอนลูกค้ามาถึงสนาม (รูดบัตร/แตะจ่าย)

เหตุผล: รองรับทั้งลูกค้าที่จองล่วงหน้าจ่ายออนไลน์ทันที และลูกค้า walk-in / คนที่สะดวกจ่ายหน้าร้านมากกว่า

## ผลกระทบต่อ booking state machine (ที่ต้องออกแบบเพิ่ม)

Flow เดิม: `pending (สร้าง PaymentIntent) → confirmed (Stripe webhook)`

ต้องเพิ่ม branch สำหรับจ่ายหน้าร้าน:
```
จองออนไลน์ → pending (hold ที่ไว้ ไม่ได้จ่าย)
    │
    ├─ จ่ายผ่าน Stripe ทันที → confirmed (เหมือนเดิม)
    │
    └─ เลือก "จ่ายหน้าร้าน" → ถึงสนาม → พนักงานรับเงินผ่าน POS2U
           → POS2U แจ้งกลับระบบเรา (ยังไม่รู้วิธี — รอ API doc)
           → confirmed
```

จุดที่ยังไม่รู้และกระทบการออกแบบ hold-expiry: ถ้าเลือกจ่ายหน้าร้าน จะ hold ที่ไว้นานแค่ไหนก่อนปล่อยคืน (กันคนจองทิ้งไว้ไม่มาจ่าย)?

## คำถามที่ต้องถาม POS2U ก่อนออกแบบต่อ

ส่งให้ทีม POS2U เพื่อขอ API documentation พร้อมตอบ 4 ข้อนี้:

1. **Webhook/callback** — เมื่อจ่ายที่เครื่องสำเร็จ POS2U แจ้งกลับระบบภายนอกอัตโนมัติได้ไหม (ไม่ใช่ต้องมีคนกดยืนยันมือ)
2. **Reference ID passthrough** — ตอนพนักงานรับเงินที่เครื่อง ใส่ "เลขที่จอง" ของเราแนบไปกับ transaction ได้ไหม (ไว้ match ว่าจ่ายรอบไหน)
3. **Query API** — เผื่อ webhook ไม่มา เรา poll เช็คสถานะ transaction ด้วย reference ได้ไหม
4. **Refund API** — ลูกค้ายกเลิก คืนเงินผ่าน API ได้ไหม หรือต้องทำมือที่เครื่อง

เพิ่มเติมที่ควรถามด้วย:
- รองรับ sandbox/test environment สำหรับ dev ไหม
- Auth model เป็นแบบไหน (API key, OAuth, IP allowlist)
- ค่าใช้จ่าย/ข้อจำกัดของ API tier (ถ้ามี)

## สิ่งที่ยืนยันจากข้อมูลสาธารณะ (ไม่ครบ ต้องยืนยันกับ POS2U อีกที)

- POS2U เป็นระบบ POS ไทย (บริษัท TechTwoYou) รองรับร้านค้าปลีก/ร้านอาหาร/ธุรกิจบริการ
- มี "ระบบ Table Order" สำหรับร้านอาหาร — เป็นการสั่งอาหารที่โต๊ะ **ไม่ใช่** ระบบจองเวลาล่วงหน้าแบบ time-slot เหมือนที่โปรเจกต์นี้ใช้
- API ที่ POS2U โฆษณาไว้ (จากหน้าเว็บสาธารณะ) เน้น "sync ข้อมูลขาย/สต๊อก/ลูกค้า/พนักงาน ออกไประบบบัญชี/ERP" — ไม่ชัดว่ารองรับ webhook แจ้งเตือน transaction แบบ real-time เข้าสู่ระบบภายนอกหรือไม่ **ต้องถามตรงเพื่อยืนยัน**

## ขั้นตอนถัดไป

1. ส่งคำถามด้านบนให้ POS2U ขอ API doc + sandbox access
2. พอได้ spec จริง → อัปเดตเอกสารนี้ด้วยรายละเอียด endpoint/auth จริง
3. ออกแบบ schema เพิ่ม (เช่น `bookings.payment_method`, `bookings.pos2u_transaction_ref`) ตาม spec ที่ได้
4. Implement webhook/poll handler + เพิ่มตัวเลือก "จ่ายหน้าร้าน" ในหน้าจองและ admin
