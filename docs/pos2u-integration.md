# POS2U Integration — สถานะและแผน

> สถานะ: **ฝั่งเรา (รับ confirm การจ่ายเงิน) สร้างเสร็จแล้ว** — เหลือรอยืนยัน spec จริงจาก POS2U ว่า query แบบไหนได้บ้าง ก่อนต่อ polling อัตโนมัติ

## การตัดสินใจที่ล็อกไว้แล้ว

**ระบบจะรับชำระเงินได้ 2 ทาง (ไม่ใช่แทนกัน):**

1. **ออนไลน์** — Stripe (PromptPay QR / บัตร) ตอนจองผ่านเว็บ/แอป → `confirmed` ทันที (ตามแผนเดิมใน `docs/brief-v2.md`)
2. **หน้าร้าน** — ผ่านเครื่อง **POS2U** ตอนลูกค้ามาถึงสนาม (รูดบัตร/แตะจ่าย)

เหตุผล: รองรับทั้งลูกค้าที่จองล่วงหน้าจ่ายออนไลน์ทันที และลูกค้า walk-in / คนที่สะดวกจ่ายหน้าร้านมากกว่า

## ⚠️ อัปเดตสำคัญ: POS2U API เป็นแบบ Pull เท่านั้น (ยืนยันจากสไลด์ทางการของ POS2U)

สไลด์ "REST API (JSON)" ของ POS2U ระบุชัดว่า:

> **ระบบภายนอกส่ง Request เข้า POS2U API → POS2U ตอบกลับเป็นข้อมูล JSON**

ตัวอย่างการใช้งานในสไลด์คือระบบบัญชี/ERP (Oracle NetSuite, Microsoft 365, SAP) **เป็นฝ่ายไปดึงข้อมูล** ยอดขาย/สต๊อก/ลูกค้า/พนักงานจาก POS2U — **ไม่มีการพูดถึง webhook หรือ POS2U ยิงแจ้งเหตุการณ์ออกมาเอง**

**ผลต่อสถาปัตยกรรม:** สมมติฐานเดิมที่ว่า "POS2U จะเรียก webhook ของเราตอนจ่ายเงินสำเร็จ" **มีแนวโน้มใช้ไม่ได้** ต้องเปลี่ยนเป็นเราเป็นฝ่าย **poll (ถาม) POS2U API เป็นระยะ** แทน — ยังต้องยืนยันกับ POS2U อีกทีว่า query ยอดขายด้วย reference/ช่วงเวลาได้จริงไหม

## สิ่งที่สร้างเสร็จแล้ว (ใช้งานได้ทันที ไม่ต้องรอ POS2U)

เพราะเราคุม schema เอง เลยสร้าง **"ฝั่งรับ" (ที่ POS2U หรือ polling job ในอนาคตจะมาเรียก)** ไว้ล่วงหน้าได้ โดยไม่ต้องรอ spec จาก POS2U:

### 1. REST API: `POST /api/payments/pos-confirm`
- Auth: header `Authorization: Bearer <POS_CONFIRM_API_KEY>`
- Body: `{ bookingId, amount, transactionRef, method? }`
- ตรวจสอบ: booking มีอยู่จริง + สถานะเป็น `pending` + ยอดตรงกับ `total` ก่อน confirm เท่านั้น (กันยอดผิด/จองซ้ำ)
- สำเร็จ → เปลี่ยน booking เป็น `confirmed` + สร้าง record ใน `payments` (`status: succeeded`)
- ใช้ Supabase **service role** client (bypass RLS) เพราะเป็นการเรียกจากระบบภายนอก ไม่มี user session
- ทดสอบแล้ว: 401 (ไม่ใส่/ใส่ผิด key), 400 (ข้อมูลไม่ครบ/JSON ผิด), 404 (ไม่พบการจอง) — ทำงานถูกต้องตามที่ออกแบบ

### 2. ปุ่ม "ยืนยันชำระแล้ว" ใน Admin Dashboard
- สำหรับการจองที่สถานะ `pending` — พนักงานกดเองหลังเห็นยอดจ่ายสำเร็จที่เครื่อง POS2U
- **ใช้งานได้ทันที ไม่ต้องพึ่ง POS2U API เลย** — เป็น fallback ที่ทำงานเสมอไม่ว่า POS2U จะรองรับ automation แค่ไหน

> หมายเหตุ: ทั้งสองส่วนนี้ยังไม่มี booking ใดใน DB ที่สร้างเป็นสถานะ `pending` เลย (ตอนนี้ `createBooking` ยัง auto-confirm ทุกรายการตามโหมด beta) — ต้องรอผูกกับตัวเลือก "จ่ายหน้าร้าน" ในหน้าจองจริง หรือกับ Stripe (ซึ่งจะสร้าง `pending` ตอนเปิด PaymentIntent เหมือนกัน) ถึงจะมีอะไรให้ทั้งสองกลไกนี้ทำงานจริง

## คำถามที่ต้องถาม POS2U (อัปเดตแล้วให้ตรงกับสิ่งที่รู้เพิ่ม)

1. **Query transaction API** — ค้นหา transaction ด้วย reference number / ช่วงเวลา / จำนวนเงินได้ไหม (นี่คือกลไกหลักที่จะใช้แทน webhook)
2. **Reference passthrough** — ตอนพนักงานรับเงินที่เครื่อง ใส่ "เลขที่จอง" ของเราแนบไปกับ transaction ได้ไหม (ไม่งั้นต้อง match ด้วยยอดเงิน+เวลาซึ่งเสี่ยงชนกัน)
3. **Rate limit / polling interval** — เรียก API ถี่แค่ไหนได้โดยไม่โดนบล็อก (จะกำหนดความถี่ polling job)
4. **Refund API** — คืนเงินผ่าน API ได้ไหม หรือต้องทำมือที่เครื่อง
5. **Auth model** — API key แบบไหน, ต้องขอผ่าน dashboard ของร้านหรือทีม POS2U ออกให้
6. **Sandbox/test environment** — มีให้ทดสอบก่อนขึ้นจริงไหม

## ขั้นตอนถัดไป

1. ส่งคำถามด้านบนให้ POS2U ขอ API doc + sandbox access
2. พอได้ spec จริง → เขียน polling job (เช่น Vercel Cron ทุก N นาที) เรียก POS2U query API แล้ว match กับ booking ที่ `pending` → เรียก `/api/payments/pos-confirm` ของเราเอง (endpoint พร้อมแล้ว)
3. เพิ่มตัวเลือก "จ่ายหน้าร้าน" ในหน้าจองลูกค้า (ตอนนี้ยังไม่มี — ทุกการจอง auto-confirm อยู่ในโหมด beta)
4. ระหว่างรอ: พนักงานใช้ปุ่ม "ยืนยันชำระแล้ว" ในแอดมินแทนได้เลย ไม่ต้องรอ POS2U
