## สถาปัตยกรรมที่ยืนยันแล้ว

ระบบมี 2 ทางเข้าที่ใช้ **DB เดียวกัน** (Supabase อันเดียว ไม่ใช่ sync คนละฐาน):

1. **Webapp ของเรา** — ลูกค้าจองออนไลน์เอง (ทางเดิม)
2. **POS2U หน้าร้าน** — พนักงานดูคิว/จองแทนลูกค้า walk-in หรือลูกค้าโทรมาจอง + รับชำระเงิน

**POS2U เป็นฝ่าย "เรียกเข้ามา" หาเรา** (client) ผ่าน REST API ที่เราเปิดให้ — เราคุม endpoint/auth/logic ทั้งหมด สิ่งที่ยังไม่รู้คือ POS2U (หรือซอฟต์แวร์หน้าร้าน) เรียก external API ได้จริงไหม — ยังต้องถาม Central Group/POS2U

### ภาพรวมง่ายๆ: 2 ทางเข้า → API กลาง → DB เดียวกัน → ออกไปอีก 2 ทาง

ดูไดอะแกรมภาพรวม (ภาษาคนอ่านง่าย ไม่ใช้ศัพท์เทคนิค) ที่: **https://www.figma.com/board/LejYdaM4gT7iDi9EenzaJ9**

| ช่องทาง | ทำหน้าที่อะไร | ใครยืนยันการจ่ายเงิน |
|---|---|---|
| **POS2U หน้าร้าน** | พนักงานดูคิว + จองแทนลูกค้า walk-in/โทรจอง, รับชำระที่เครื่อง POS2U หน้างาน | POS2U เป็นคนยืนยัน แล้วแจ้งเรามาตรงๆ |
| **WebApp + 2C2P** | ลูกค้าจองเองผ่านเว็บ/แอป, จ่ายเงินออนไลน์ผ่าน 2C2P | 2C2P เป็นคนยืนยัน แล้วยิง webhook แจ้งเรามาตรงๆ |

ทั้งสองทาง **ไม่คุยกันเอง** — คุยกับ **API กลางของเรา** อย่างเดียว แล้ว API กลางนี้เขียน/อ่านจาก **DB เดียวกัน** (Supabase) เสมอ ไม่ว่าการจองจะมาจากฝั่งไหน (สนามไหนก็ตาม — สนาม 1, สนาม 2, ... ทุกสาขา) เราจึงเห็นคิวสนามที่ตรงกันทั้งสองทางแบบ real-time โดยไม่ต้อง sync ข้อมูลข้ามระบบ

จาก DB เดียวกันนี้ แตกออกไป 2 ทาง:
1. **สถานะการจอง** — ทุกการจองไล่ผ่านสถานะเหล่านี้: **จองไว้ยังไม่จ่าย (Reserve)** → **จ่ายเงินแล้ว (Booked/Paid)** → แล้วอาจไปต่อที่ **ยกเลิก (Cancel)** หรือ **คืนเงิน (Refund)** — ไม่ว่าจะจองผ่าน POS2U หรือ WebApp ก็ไล่สถานะชุดเดียวกันนี้
2. **Dashboard แอดมิน** — อ่านจาก DB เดียวกันนี้เหมือนกัน ทำให้เห็นคิวสนามสดและยอดรายรับอัพเดทอัตโนมัติทันทีที่มีการจอง/จ่ายเงิน ไม่ว่าจะมาจากช่องทางไหน

## คำถามที่ 1: ดูคิวเสร็จแล้ว ต้อง "จอง/hold" ก่อนจ่ายไหม?

**ตอบ: ต้อง hold ก่อนเสมอ** — จองแบบ `pending` ทันทีที่พนักงานเลือกช่องให้ลูกค้า **ก่อน**ไปรับเงินที่เครื่อง ไม่ใช่รอจ่ายเสร็จแล้วค่อยสร้าง booking (ตรงนี้ยังเหมือนเดิม — ไม่เปลี่ยน)

**เหตุผล:** ถ้าไม่ hold ก่อน จะมีช่องว่างระหว่าง "ลูกค้าเลือกเวลา" กับ "จ่ายเงินเสร็จ" ที่คนอื่นจองเวลาเดียวกันแทรกได้ — นี่คือ race condition ที่ถามในข้อ 2 (เรื่องนี้แยกจากเรื่องสถานะการจ่ายเงินด้านล่าง — สร้าง booking กันชนกัน ≠ ใครเป็นคนบอกว่าจ่ายแล้ว)

### แก้ไข: เราไม่ต้องเป็นคนจัดการ "หมดเวลา hold" เอง

หลังจากคุยกันใหม่ — **เราแค่รับสถานะที่ POS2U บอกมา ไม่ต้องมี logic ประเมิน/รอ/นับเวลาเอง**:

- POS2U ส่ง payload บอกสถานะการจองมาให้เราตรงๆ เช่น `reserved` (จองไว้แล้วยังไม่จ่าย — ใช้กรณี event จองล่วงหน้า จ่ายหน้างาน) หรือ `paid` (จ่ายแล้วที่เครื่อง POS2U)
- เรามีหน้าที่แค่ **รับและอัปเดตสถานะตามที่บอกมา** ไม่ต้องรอประมวลผลหรือตัดสินใจเอง — `reserved` → เก็บเป็น `bookings.status = 'pending'` (enum เดิมมีอยู่แล้ว ไม่ต้องเพิ่มค่าใหม่), `paid` → อัปเดตเป็น `confirmed` + insert แถว `payments` (เหมือนที่ `pos-confirm` ทำอยู่ตอนนี้)
- **`hold_expires_at` + cron ปล่อยคืน slot อัตโนมัติ → ตัดออกจาก scope นี้** ไม่ใช่สิ่งที่เราต้องสร้าง/ดูแลเอง ถ้า POS2U หรือพนักงานยกเลิก hold เอง เขาจะ PATCH บอกเรา (`status = cancelled`) เราก็แค่รับมาอัปเดต — คอลัมน์ `hold_expires_at` ทิ้งไว้เฉยๆ ได้ (อาจใช้โชว์ในแอดมินภายหลังว่า "ค้างชำระมา X นาทีแล้ว" แต่ไม่ผูกกับ logic อัตโนมัติใดๆ)

**นี่คือแพทเทิร์นเดียวกับการจ่ายผ่านเว็บแอปด้วย Stripe/2C2P** — gateway เป็นคนยืนยันว่าจ่ายสำเร็จแล้วค่อยยิง webhook บอกเรา เราไม่ได้ไปเช็คหรือรอผลเอง เป็น pattern เดียวกันทั้งสองทาง (POS2U ฝั่งหน้าร้าน, Stripe/2C2P ฝั่งออนไลน์) — เราเป็นแค่ "ผู้รับข้อมูลที่ยืนยันแล้ว" ทั้งสองทาง

### เพิ่ม: แยกแหล่งที่มาของการจ่ายเงิน (`payment_source`)

เพื่อให้ dashboard/รายงานแยกได้ว่ารายได้มาจากช่องทางไหน — เพิ่มคอลัมน์ `payments.source` (text: `'pos2u' | 'stripe' | '2c2p' | 'manual_admin'`) แยกจาก `payments.method` (วิธีจ่ายจริง เช่น `'card' | 'promptpay' | 'cash'` ที่ POS2U รายงานมา) — เป็น migration เล็กๆ ไม่กระทบของเดิม จะทำ `.sql` ให้รันทีหลังตอนพร้อมสร้าง endpoint จริง

## คำถามที่ 2: ป้องกัน race condition ยังไง

แยกตามประเภทเพราะกลไกต่างกัน:

### จองเหมาคอร์ท (private) — **มีการป้องกันอยู่แล้ว ✅**
Schema มี PostgreSQL EXCLUDE constraint (`bookings_no_private_overlap`) ล็อกที่ระดับ database — กันคอร์ท+ช่วงเวลาเดียวกันถูกจองซ้ำแบบ atomic ไม่ว่าจะยิงมาจากเว็บ, POS2U, หรือที่ไหนก็ตาม ถ้าชนกัน Postgres จะ reject การ insert ที่สองเองทันที ไม่ต้องเขียน lock เพิ่ม

### Open Play (นับที่นั่งตาม capacity) — **⚠️ ยังไม่มีการป้องกันเลย (ช่องโหว่จริงที่ใช้งานอยู่ตอนนี้)**
เช็คโค้ด `createBooking` (ที่เว็บใช้อยู่ปัจจุบัน) แล้ว **ไม่มีการเช็ค capacity ก่อน insert เลย** — insert ตรงๆ โดยไม่ล็อกแถว session ก่อน หมายความว่า **ถ้ามี 2 คนกดจองรอบ Open Play ที่เหลือที่นั่งสุดท้ายพร้อมกัน ระบบจะปล่อยให้จองเกิน capacity ได้จริงในตอนนี้** — ไม่ใช่แค่เรื่องใหม่ที่เกี่ยวกับ POS2U แต่เป็นบั๊กที่มีอยู่แล้วในเว็บด้วย

**วิธีแก้ที่ถูกต้อง:** เพิ่ม **DB trigger** (`BEFORE INSERT` บน `bookings` เฉพาะแถว `booking_type = 'open_play'`) ที่ lock แถว session (`SELECT ... FOR UPDATE`) แล้วเช็ค `seats_taken + seats <= capacity` ก่อนอนุญาตให้ insert — ทำที่ระดับ database เหมือนกับ private เพื่อป้องกันไม่ว่า insert จะมาจากไหน (เว็บ, POS2U POST, admin) โดยไม่ต้องพึ่งให้ทุก endpoint จำไปเช็คเอง

## คำถามที่ 3: Refund

Schema มี `payments.status` และ `bookings.status` รองรับค่า `refunded` อยู่แล้ว (ไม่ต้อง migrate เพิ่ม) — ไม่มีตาราง `refunds` แยก (มีแค่ในเอกสารแผนเดิม ไม่เคยสร้างจริง) ใช้ field เดิมพอสำหรับ MVP

**Flow:** ลูกค้าขอยกเลิกหลังจ่ายแล้ว → พนักงาน reverse รายการที่เครื่อง POS2U → POS2U (หรือพนักงานเอง) แจ้งกลับเราผ่าน `PATCH /api/payments/pos-refund` (ยังไม่สร้าง) → เปลี่ยน `bookings.status = refunded`, `payments.status = refunded`

## Walk-in customer identity — ตัดสินใจแล้ว

เลือก **shadow account จากเบอร์โทร**: `POST /api/pos/bookings` รับ `customerPhone` (+ `customerName` ไม่บังคับ) แล้วระบบค้นหา/สร้างบัญชีให้อัตโนมัติ (`src/lib/pos-customer.ts`) — เบอร์เดียวกันจะโยงเป็นลูกค้าคนเดียวกันทั้งจองผ่าน POS2U และเว็บ ทำให้ CRM เห็นประวัติครบ ไม่ว่าจะมาช่องทางไหน ถ้าลูกค้าคนนั้น login ผ่านเว็บด้วยเบอร์เดียวกันภายหลัง ก็จะเจอบัญชีเดิมทันที

## งานที่ทำไปแล้ว

1. ✅ **DB trigger กันจองเกิน capacity ของ Open Play** — แก้บั๊กที่ใช้งานอยู่จริงในเว็บ ไม่ใช่แค่เรื่อง POS2U (`docs/migration-pos2u.sql`)
2. ✅ **Migration**: เพิ่ม `payments.source`, `bookings.channel`, unique index บน `profiles.phone`
3. ✅ **ตัดสินใจ walk-in customer identity** → shadow account จากเบอร์โทร
4. ✅ สร้าง `POST /api/pos/bookings`, `PATCH /api/pos/bookings/:id`, `PATCH /api/payments/pos-refund`
5. ✅ เพิ่ม `source` ใน `PATCH /api/payments/pos-confirm`
6. ✅ เขียน API doc เต็ม (`docs/pos2u-api-doc.md`)

## งานที่เหลือ

1. **รัน `docs/migration-pos2u.sql` บน Supabase production** — ยังไม่ได้รัน โค้ดฝั่งแอปพลิเคชันจะพังถ้ารันก่อนไม่ครบ (คอลัมน์/trigger ยังไม่มี)
2. ถามคำถามที่ค้างกับ Central Group/POS2U: เรียก external API ของเราได้จริงไหม (จากที่เห็น marketing slide ก่อนหน้านี้ POS2U ดูจะเป็นฝั่งเปิด API ให้คนอื่นเรียก ไม่ใช่ตัวเรียกออก — ต้องถามให้ชัดเจน), auth model ฝั่งเขารองรับ static Bearer token ไหม
3. ตั้ง env vars บน production ก่อนใช้งานจริง: `SUPABASE_SERVICE_ROLE_KEY`, `POS_API_KEY`
4. Test end-to-end กับ POS2U จริงยังไม่ได้ทำ (ทดสอบได้แค่ auth/validation logic ในนี้ เพราะไม่มี DB จริงให้ต่อ)
